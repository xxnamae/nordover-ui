Nordover Design System

Universal foundation of tokens and building blocks for web and app interfaces. Every component below is shipped in components-web.css / components-app.css. Toggle the package switcher to preview web vs. app token defaults.

App vs. Web — Which Package?

Nordover is two complete systems with identical architecture but different defaults. Choose one based on your project type.

App Package

For dashboards, SaaS tools, and work applications

Characteristics:
  • Statisk typografi (14px base, kompakt)
  • Tett spacing (1rem gaps)
  • Dark mode som default
  • Brand-fiolett accent
  • Tactile buttons (elevated + gradient)
Velg App hvis du bygger:
  • Dashboard eller admin-grensesnitt
  • Arbeidsverktøy eller SaaS
  • Data-intenst grensesnitt

Web Package

For websites, marketing pages, and editorial content

Characteristics:
  • Fluid typografi (16px base, luftig)
  • Romslig spacing (1.5rem gaps)
  • Light mode som default
  • Nøytral sort accent
  • Flat buttons som default
Velg Web hvis du bygger:
  • Nettside eller landing page
  • Marketing-kampanje
  • Innholds- eller nyhetsnettsted
Key Principle:

Samme klasse → samme token-navn i begge pakker. Verdiene er bare ulike. Eksempel: .t-body-lg bruker --text-md i begge, men verdien er 16px (app) vs. clamp-fluid (web).

Usikker på valg?

Start med App — det er det mest universelle valget og fungerer både for interne og eksterne grensesnitt. Web-pakken er optimalisert for lesing og innholdsorienterte prosjekter.

Web or App package?
  • web → marketing / editorial: airy spacing, large fluid type
  • app → dashboard / SaaS: compact spacing, denser type
Good to know
  • Tokens travel, components don't. Elementor generates its own markup — adopt the tokens via Global Colors, Fonts, and var(--…) references.
  • v4 JSON limits: colors are pre-resolved hex; shadows and dark-mode aren't included — add via Custom CSS using the token block in Documentation.
  • Light / dark: for marketing sites pick one mode; for dark sections swap --color-bg / --color-fg locally.
  • Token updates: re-download and re-import — new values replace old ones everywhere.

Buttons

Triggers an action or navigation. One .btn base with style variants, sizes, and states — pick the variant by emphasis, not by colour.

.btn.btn-primary.btn-secondary .btn-ghost.btn-tonal.btn-elevated .btn-link.btn-error.btn-sm .btn-lg.btn-touch
When to use

Use for

  • Committing an action (Save, Submit, Delete)
  • One primary button per view to signal the main path
  • .btn-error for destructive actions, paired with confirmation

Avoid for

  • Plain navigation between pages — use a link or .btn-link
  • Multiple competing primaries in the same area
  • Long sentences — labels should be a verb or verb + noun

Badges & Alerts

Two status patterns sharing one colour vocabulary. A badge labels a single item; an alert communicates a message about the current context.

.badge.badge-success.badge-error .badge-warning.badge-info.badge-neutral .alert.alert-success.alert-icon
When to use

Use for

  • Badge — a compact status on a row, card, or tab (e.g. “Active”, “3”)
  • Alert — an inline message about a form, page, or section
  • The same status words everywhere: success / error / warning / info

Avoid for

  • Transient confirmations that should disappear — use a Toast
  • Badges as buttons — they aren't interactive
  • Walls of alerts; show the most important one
