Nordover Design System

Universal foundation of tokens and building blocks for web and app interfaces. Every component below is shipped in components-web.css / components-app.css. Toggle the package switcher to preview web vs. app token defaults.

Getting Started — Elementor Integration

Use Nordover in your WordPress site. Two paths: Elementor v3 (manual setup) or v4 (automatic). Pick your version below.

Elementor v3 (Classic Editor)

Manual mapping — 30 min setup

/* Step 1: Paste tokens in Site Settings → Custom CSS */ :root { --color-accent: #060709; --color-fg: #1a1a1a; --space-4: 1rem; /* ... all 254 tokens ... */ }
  1. Paste all tokens from tokens-web.css into Site Settings → Custom CSS
  2. Map colors manually: Elementor → Dashboard → Global Colors, create each one with var(--color-*)
  3. Map fonts: Global Fonts, reference --font-sans, --font-display
  4. Build: Use Elementor widgets; add var(--space-4), var(--radius-md) in Custom CSS fields

Elementor v4 (Editor V4) ⭐ Recommended

JSON import — 5 min setup

/* No manual work needed! Import the JSON, */ /* all 254 tokens appear in every picker. */ ✓ Download: nordover-elementor-v4-web.json
  1. Download nordover-elementor-v4-web.json (or v4-app.json for dense dashboards)
  2. Import: Elementor Dashboard → Kits → Tools → Import (or drag file)
  3. Regenerate: Tools → Regenerate CSS & Data
  4. Build: All tokens now in color/typography/sizing pickers automatically
Which package?
  • v4-web.json or tokens-web.css → marketing/editorial sites (airy spacing, large type)
  • v4-app.json or tokens-app.css → app-like dashboards (compact, dense)

See docs/handoff/ELEMENTOR-WORDPRESS.md for detailed steps, gotchas, and light/dark mode setup.

Buttons

Triggers an action or navigation. One .btn base with style variants, sizes, and states — pick the variant by emphasis, not by colour.

.btn.btn-primary.btn-secondary .btn-ghost.btn-tonal.btn-elevated .btn-link.btn-error.btn-sm .btn-lg.btn-touch
When to use

Use for

  • Committing an action (Save, Submit, Delete)
  • One primary button per view to signal the main path
  • .btn-error for destructive actions, paired with confirmation

Avoid for

  • Plain navigation between pages — use a link or .btn-link
  • Multiple competing primaries in the same area
  • Long sentences — labels should be a verb or verb + noun

Guidelines
Do

One primary per group; pair it with a lower-emphasis secondary for the alternative.

Don't

Don't make two primaries compete — the user can't tell which path is the default.

Accessibility
  • Use a real <button> (or <a> for navigation) so keyboard and screen-reader support is free.
  • While loading, set aria-busy="true"; while disabled, the native disabled attribute removes it from the tab order.
  • Icon-only buttons need an aria-label; use .btn-touch to guarantee a 44px target.
Code
<button class="btn btn-primary">Save</button>
<button class="btn btn-secondary">Cancel</button>
<button class="btn btn-error">Delete</button>
<button class="btn btn-primary btn-touch" aria-busy="true">Saving…</button>

Forms & Inputs

Text inputs, textareas, selects, checkboxes, radios, switches, plus label/help/error helpers.

.field.field-label.form-input .form-textarea.form-select.form-checkbox .form-radio.form-switch.field-help .field-error.form-group.form-group-item
When to use

Use for

  • Collecting input — always wrap a control in .field with a .field-label
  • .field-help for guidance, .field-error for validation messages
  • .form-switch for instant on/off; checkboxes for multi-select

Avoid for

  • Placeholder text as the only label — it disappears on input
  • A switch where the change isn't applied immediately — use a checkbox
  • Colour alone to signal errors — pair with .field-error text

Add .is-dragover to .file-upload while a drag is over the dropzone (set it from your dragenter/dragleave handlers) to surface the active accent border and tint.

When to use

Use for

  • Uploading one or more files with drag-and-drop or click
  • Showing per-file progress and a removable list

Avoid for

  • A single avatar/logo swap — a plain file button is enough
  • Hiding accepted types/limits — always state them up front

  • The dropzone is a <label> wrapping a real <input type="file"> — click and keyboard work natively.

Badges & Alerts

Two status patterns sharing one colour vocabulary. A badge labels a single item; an alert communicates a message about the current context.

.badge.badge-success.badge-error .badge-warning.badge-info.badge-neutral .alert.alert-success.alert-icon
When to use

Use for

  • Badge — a compact status on a row, card, or tab (e.g. “Active”, “3”)
  • Alert — an inline message about a form, page, or section
  • The same status words everywhere: success / error / warning / info

Avoid for

  • Transient confirmations that should disappear — use a Toast
  • Badges as buttons — they aren't interactive
  • Walls of alerts; show the most important one

When to use

Use for

  • Brief, self-dismissing feedback after an action (“Saved”)
  • Non-blocking results the user doesn't need to act on

Avoid for

  • Critical errors that need a decision — use an inline alert or modal
  • Content the user must read fully — toasts disappear

When to use

Use for

  • First-run states, empty search results, cleared inboxes
  • Pairing the message with one clear action to move forward

Avoid for

  • Error states — those need an alert, not an empty state
  • Loading — use a skeleton while data is in flight
